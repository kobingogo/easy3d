# easy3d Phase 2A Product Spec: 批量上新工作台

**Date:** 2026-03-23  
**Status:** Draft for review  
**Owner:** Codex + user

## 1. 目标

在 Phase 1「单任务素材包」能力已打通的基础上，Phase 2A 聚焦一个更直接的商业价值目标：

> 让中小卖家一次性处理多个 SKU 的素材任务，把“单次可用”升级为“可重复上新”。

Phase 2A 的核心不是新增更复杂的 AI 能力，而是把现有能力组织成稳定、可控、可追踪的批量工作流。

---

## 2. 定位与边界

## 2.1 Phase 2A 是什么

Phase 2A 定义为：

1. 批量上传商品图（一个批次多个商品）
2. 批量生成（受控并发队列）
3. 批量查看状态（批次 + 子任务）
4. 批量交付（按批次下载素材包集合）

## 2.2 Phase 2A 不做什么

为保证交付速度和聚焦度，本阶段明确不做：

1. 用户可编辑模板（属于 Phase 2B）
2. 品牌资产库（属于 Phase 2B）
3. 完整自动发布到平台（属于 Phase 3）
4. 复杂订阅计费系统（继续沿用轻量人工/单次确认）

---

## 3. 目标用户与场景

## 3.1 目标用户

1. 1-10 人中小卖家团队
2. 小型内容电商工作室
3. 高频补款/上新的运营角色

## 3.2 核心场景

以 `包袋 / 小皮具` 为主：

1. 同一批次上新 5-20 个 SKU
2. 每个 SKU 需要淘宝/小红书/抖音三端素材
3. 希望先看批次进度，再分批处理异常项

---

## 4. 产品需求（Phase 2A）

## 4.1 批量上传

1. 用户可创建一个 `批次任务`，输入批次名称和渠道目标（默认三平台）
2. 支持一次上传多张商品图（单图模式为主）
3. 每张图生成一个 `批次子任务`（对应一条模型记录）
4. 上传后立即进入队列，而不是逐条手动点击生成

建议上限（可配置）：

1. 单批次最多 20 个商品
2. 单文件上限继续沿用 Phase 1（10MB）

## 4.2 队列与调度

1. 批次级状态：`queued / running / partial_failed / completed / canceled`
2. 子任务状态：`queued / processing / completed / failed / skipped`
3. 受控并发执行，默认并发 3（可配置）
4. 失败子任务允许手动重试，不影响已完成子任务

## 4.3 批量结果与导出

1. 批次详情页提供进度总览（总数、完成数、失败数、剩余数）
2. 子任务可单独查看结果（复用 Phase 1 结果页）
3. 支持批量下载已完成子任务的素材包集合（zip of zips 或平铺目录）
4. 未完成/失败子任务要明确标记，不可静默丢失

## 4.4 与 Phase 1 的衔接

1. 单个子任务的生成链路完全复用 Phase 1
2. 单个子任务的解锁与交付状态逻辑继续复用 `deriveUnlockView(...)`
3. 批量功能只是“组织和调度层”，不重写底层生成实现

---

## 5. 体验要求

## 5.1 主流程

主流程固定为：

1. 新建批次
2. 批量上传
3. 查看队列进度
4. 处理失败项（可选重试）
5. 批量下载已完成素材包

## 5.2 信息架构

新增或重构三个页面层：

1. `/dashboard` 增加批次视图（默认入口）
2. `/dashboard/batches/:id` 批次详情（子任务列表）
3. `/generate` 保留单任务入口（作为补充，不再承担批量主入口）

## 5.3 可观察性

1. 批次和子任务状态必须可解释
2. 每个失败项给出可读错误摘要
3. 重试后保留历史状态轨迹（至少保留失败次数与最近错误）

---

## 6. 数据与系统设计

## 6.1 新增实体

新增两张核心表：

1. `batch_jobs`
2. `batch_items`

`batch_jobs` 建议字段：

1. `id`
2. `name`
3. `category`（Phase 2A 仍限定 `bags`）
4. `status`
5. `total_count`
6. `queued_count`
7. `processing_count`
8. `completed_count`
9. `failed_count`
10. `created_at / updated_at`
11. `started_at / completed_at / canceled_at`

`batch_items` 建议字段：

1. `id`
2. `batch_job_id`
3. `model_id`（关联既有 `models`）
4. `source_image_url`
5. `status`
6. `attempt_count`
7. `last_error`
8. `trip_task_id`
9. `locked_at`
10. `created_at / updated_at`

## 6.2 状态机（子任务）

1. `queued -> processing`
2. `processing -> completed`
3. `processing -> failed`
4. `failed -> queued`（重试）
5. 任意状态 -> `skipped`（取消时可选）

## 6.3 调度策略

Phase 2A 采用“轻量应用层队列”：

1. API 侧按并发上限领取 `queued` 子任务
2. 通过 `locked_at` + 状态条件更新避免重复领取
3. 领取后复用现有 `generate-smart + tripo/status` 链路
4. 完成后回写 `batch_items` 与 `batch_jobs` 聚合计数

这个方案的目标是快速上线可用版本，避免 Phase 2A 引入重型消息队列基础设施。

---

## 7. API 范围（Phase 2A）

建议新增：

1. `POST /api/batches`
   - 创建批次与子任务
2. `GET /api/batches`
   - 查询批次列表
3. `GET /api/batches/:id`
   - 查询批次详情与子任务状态
4. `POST /api/batches/:id/process`
   - 推进队列执行（受控并发）
5. `POST /api/batches/:id/items/:itemId/retry`
   - 重试失败项
6. `GET /api/batches/:id/download`
   - 下载批量交付包（仅完成项）

---

## 8. 指标与验收

## 8.1 业务指标

Phase 2A 关注指标：

1. 批次创建成功率
2. 子任务完成率
3. 批次导出使用率
4. 批量流程的二次复用率

## 8.2 体验指标

建议目标：

1. 10 个 SKU 批次在 40 分钟内完成（不含人工审核等待）
2. 批次页首次可见进度时间 < 5 秒
3. 失败项可重试成功率 > 60%

## 8.3 发布验收条件

满足以下条件才可标记 Phase 2A 完成：

1. 批量上传与批次创建可用
2. 队列并发执行稳定，无重复领取
3. 批次与子任务状态可追踪
4. 失败项重试可用
5. 批量下载可用（完成项）
6. 文档与运维手册同步更新

---

## 9. 风险与缓解

1. 并发状态竞争导致重复处理  
缓解：领取任务时使用条件更新 + lock 过期机制

2. 批次过大导致前端体验退化  
缓解：限制单批次上限，分页展示子任务

3. 批量导出体积过大  
缓解：先支持“完成项导出”，并设置导出文件数上限

4. 范围膨胀（把模板系统提前做进来）  
缓解：明确模板系统属于 Phase 2B，不在 2A 交付条件内

---

## 10. 与后续阶段的衔接

Phase 2A 完成后，Phase 2B 可以在其基础上增加：

1. 用户可保存模板
2. 品牌风格复用
3. 批次级模板绑定策略

Phase 2A 的成功标准是“批次可跑通”，而不是“模板功能完整”。
